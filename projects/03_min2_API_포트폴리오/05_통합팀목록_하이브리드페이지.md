## 🎯 문제 정의

### 상황: 운영팀의 통합 대시보드

```
요구사항:
✓ 여러 연도(2024, 2025, 2026) 통합 조회
✓ 여러 트랙(도전기업, 펠로우, ...) 동시 조회
✓ 여러 필터(지역, 아이템 분야, ...)
✓ 여러 정렬(서류심사 결과, 발표 결과, ...)
✓ 팀원 정보 표시
✓ 페이지네이션 (10개/20개/50개 선택)

→ 데이터 규모: 연도별 트랙당 500~1000팀 × 3년 × 5트랙 = 수만 팀
```

### 복잡성

```
❌ 문제 1: 심사결과는 DB 컬럼이 아님
   └─ stage_choice 테이블과의 조합으로 계산 필드

❌ 문제 2: 스테이지 구조가 트랙마다 다름
   ├─ 어떤 트랙: 서류만
   ├─ 어떤 트랙: 서류 + 발표
   └─ 어떤 트랙: 서류 + 발표 + 최종
   → 모든 트랙을 통합 쿼리하면 LEFT JOIN 폭발

❌ 문제 3: 심사결과로 정렬하려면?
   └─ DB에서 CASE WHEN으로 할 수 없음 (동적으로 계산)
   └─ 메모리에서 정렬해야 함
   → 전체 데이터를 메모리에 로드해야 함 ❌

❌ 문제 4: 일반 필드로 정렬하면서 페이징하려면?
   └─ DB LIMIT/OFFSET으로 충분
   → 계산 필드 정렬 때만 전체 로드? 성능 차이 큼!
```

### 기존 시도의 문제

```
시도 1: DB에서 모든 계산
└─ OUTER JOIN × 트랙별 스테이지
└─ 쿼리가 기하급수적으로 복잡해짐
└─ 실패 ❌

시도 2: 전체 데이터를 항상 메모리로 로드
└─ 수만 팀의 정보를 매번 메모리에 올림
└─ 힙 메모리 부담, GC 증가
└─ 느려짐 ❌

시도 3: N+1 쿼리 (팀마다 스테이지 조회)
└─ 수천 쿼리 발생
└─ 느려짐 ❌
```

## 💡 솔루션: 하이브리드 페이지네이션

```
요청 분석:
┌──────────────────────────────────────────┐
│ sortField ∈ [documentResult, ...] ?      │
│ hasResultFilter ? (필터에 계산필드 있나?) │
└──────────────────────────────────────────┘
    │
    ├─ YES (계산필드 관련) 
    │  └─ fullFetch = true
    │     └─ 전체 조회 → 서비스 레벨 페이징
    │
    └─ NO (일반필드만)
       └─ fullFetch = false
          └─ DB LIMIT/OFFSET

→ 경로를 동적으로 선택!
```

## 🔥 핵심 구현 코드

```java
@Transactional(readOnly = true)
public PageResponse<PointTeamUnifiedListResVO> pointTeamUnifiedList(
        PointTeamUnifiedListReqVO reqVO) {
    
    int page = reqVO.getPage();
    int pageSize = reqVO.getPageSize();
    reqVO.setOffset((page - 1) * pageSize);

    // ───────────────────────────────────────────────────────
    // [Step 1] 정렬 필드 화이트리스트 검증 (SQL Injection 방지)
    // ───────────────────────────────────────────────────────
    List<String> allowedSortFields = List.of(
        "trackYear", "tmRegion", 
        "documentResult",      // ← 계산 필드
        "presentationResult",  // ← 계산 필드
        "finalResult"          // ← 계산 필드
    );
    
    if (reqVO.getSortField() != null 
        && !allowedSortFields.contains(reqVO.getSortField())) {
        reqVO.setSortField(null); // 부적절한 필드는 NULL
    }

    // ───────────────────────────────────────────────────────
    // [Step 2] 하이브리드 경로 결정
    // ───────────────────────────────────────────────────────
    
    // 계산 필드 정렬?
    boolean isResultSort = reqVO.getSortField() != null
        && List.of("documentResult", "presentationResult", "finalResult")
               .contains(reqVO.getSortField());
    
    // 계산 필드 필터?
    boolean hasResultFilter = 
        (reqVO.getDocumentResults() != null && !reqVO.getDocumentResults().isEmpty())
        || (reqVO.getPresentationResults() != null && !reqVO.getPresentationResults().isEmpty())
        || (reqVO.getFinalResults() != null && !reqVO.getFinalResults().isEmpty());
    
    // 둘 중 하나라도 있으면 전체 로드 필요
    boolean needsFullFetch = isResultSort || hasResultFilter;

    // ───────────────────────────────────────────────────────
    // [Step 3] 경로 선택에 따라 VO 재구성
    // ───────────────────────────────────────────────────────
    
    PointTeamUnifiedListReqVO queryReqVO = reqVO;
    
    if (needsFullFetch) {
        // 페이징 해제 (전체 조회)
        queryReqVO = PointTeamUnifiedListReqVO.builder()
            .trackYears(reqVO.getTrackYears())
            .trackNos(reqVO.getTrackNos())
            .teamNo(reqVO.getTeamNo())
            .regions(reqVO.getRegions())
            .teamItemParts(reqVO.getTeamItemParts())
            .teamName(reqVO.getTeamName())
            // ... (다른 필터들)
            .page(1)
            .pageSize(Integer.MAX_VALUE)  // ← 페이징 해제
            .offset(0)
            .build();
    }

    // ───────────────────────────────────────────────────────
    // [Step 4] DB 조회 (일반 또는 전체)
    // ───────────────────────────────────────────────────────
    
    ArrayList<PointTeamUnifiedListResVO> teamList = 
        pointStageQueryMapper.pointTeamUnifiedList(queryReqVO);
    
    if (teamList.isEmpty()) {
        return emptyPage(page, pageSize);
    }

    // ───────────────────────────────────────────────────────
    // [Step 5] 심사결과 계산 후 주입
    // ───────────────────────────────────────────────────────
    
    List<Long> trackNosForEval = teamList.stream()
        .map(PointTeamUnifiedListResVO::getTrackNo)
        .distinct()
        .toList();
    
    setEvaluationResults(teamList, trackNosForEval);
    // → 각 팀에 documentResult, presentationResult, finalResult 주입됨

    // ───────────────────────────────────────────────────────
    // [Step 6-A] 계산 필드 필터링 (IN-방식 복수값)
    // ───────────────────────────────────────────────────────
    
    if (hasResultFilter) {
        // 서류 결과 필터
        if (reqVO.getDocumentResults() != null 
            && !reqVO.getDocumentResults().isEmpty()) {
            teamList.removeIf(t -> 
                t.getDocumentResult() == null
                || !reqVO.getDocumentResults().contains(t.getDocumentResult()));
        }
        
        // 발표 결과 필터 (동일 패턴)
        if (reqVO.getPresentationResults() != null 
            && !reqVO.getPresentationResults().isEmpty()) {
            teamList.removeIf(t -> 
                t.getPresentationResult() == null
                || !reqVO.getPresentationResults().contains(t.getPresentationResult()));
        }
        
        // 최종 결과 필터
        if (reqVO.getFinalResults() != null 
            && !reqVO.getFinalResults().isEmpty()) {
            teamList.removeIf(t -> 
                t.getFinalResult() == null
                || !reqVO.getFinalResults().contains(t.getFinalResult()));
        }
    }

    // ───────────────────────────────────────────────────────
    // [Step 6-B] 계산 필드 기준 정렬 (Comparator switch)
    // ───────────────────────────────────────────────────────
    
    if (isResultSort) {
        boolean isAsc = !"DESC".equalsIgnoreCase(reqVO.getSortDirection());
        
        Comparator<PointTeamUnifiedListResVO> cmp = switch (reqVO.getSortField()) {
            case "documentResult" -> 
                Comparator.comparing(PointTeamUnifiedListResVO::getDocumentResult,
                    Comparator.nullsLast(
                        isAsc ? Comparator.naturalOrder() : Comparator.reverseOrder()));
            
            case "presentationResult" -> 
                Comparator.comparing(PointTeamUnifiedListResVO::getPresentationResult,
                    Comparator.nullsLast(
                        isAsc ? Comparator.naturalOrder() : Comparator.reverseOrder()));
            
            case "finalResult" -> 
                Comparator.comparing(PointTeamUnifiedListResVO::getFinalResult,
                    Comparator.nullsLast(
                        isAsc ? Comparator.naturalOrder() : Comparator.reverseOrder()));
            
            default -> 
                Comparator.comparing(PointTeamUnifiedListResVO::getTrackYear);
        };
        
        teamList.sort(cmp);
    }

    // ───────────────────────────────────────────────────────
    // [Step 7] 경로별 페이징 처리 분기
    // ───────────────────────────────────────────────────────
    
    int totalElements = teamList.isEmpty() ? 0 : teamList.get(0).getTotalElements();
    
    if (needsFullFetch) {
        // 서비스 레벨 페이징 (in-memory subList)
        int from = Math.min((page - 1) * pageSize, totalElements);
        int to   = Math.min(from + pageSize, totalElements);
        
        // 페이징된 리스트
        if (from < totalElements) {
            teamList = new ArrayList<>(teamList.subList(from, to));
        } else {
            teamList = new ArrayList<>();
        }
        
        totalElements = teamList.isEmpty() ? 0 : teamList.get(0).getTotalElements();
        
    } else {
        // DB 페이징 (이미 LIMIT/OFFSET 적용됨, totalElements는 DB에서)
        totalElements = teamList.isEmpty() ? 0 : teamList.get(0).getTotalElements();
    }

    // ───────────────────────────────────────────────────────
    // [Step 8] 추가 정보 로드 (팀원, 순번)
    // ───────────────────────────────────────────────────────
    
    setMemberList(teamList);        // 현재 페이지 팀원만 조회
    setRowNoAndTeamSeq(teamList, (page - 1) * pageSize);

    return PageResponse.of(teamList, page, pageSize, totalElements);
}
```

### Helper: `setMemberList` (N+1 회피)

```java
private void setMemberList(List<PointTeamUnifiedListResVO> teamList) {
    if (teamList.isEmpty()) return;
    
    // 현재 페이지의 팀 번호만 모아서
    List<Long> teamNos = teamList.stream()
        .map(PointTeamUnifiedListResVO::getTeamNo)
        .toList();
    
    // IN-query 1회로 모든 팀원 조회
    List<TeamMemberVO> allMembers = pointStageQueryMapper
        .selectTeamMembersByTeamNos(teamNos);
    
    // 팀별 멤버 그룹화
    Map<Long, List<TeamMemberVO>> membersByTeam = allMembers.stream()
        .collect(Collectors.groupingBy(TeamMemberVO::getTeamNo));
    
    // 팀 객체에 주입
    teamList.forEach(team -> 
        team.setMembers(membersByTeam.getOrDefault(team.getTeamNo(), 
                                                   Collections.emptyList())));
}
```

### Helper: `setRowNoAndTeamSeq` (페이지 경계 처리)

```java
private void setRowNoAndTeamSeq(List<PointTeamUnifiedListResVO> teamList, int startRow) {
    Map<Long, Integer> teamSeqMap = new LinkedHashMap<>();
    int rowNo = startRow + 1;
    
    for (PointTeamUnifiedListResVO team : teamList) {
        team.setRowNo(rowNo++);
        
        // 같은 팀이 여러 트랙에 있을 수 있음
        if (!teamSeqMap.containsKey(team.getTeamNo())) {
            teamSeqMap.put(team.getTeamNo(), teamSeqMap.size() + 1);
        }
        team.setTeamSeq(teamSeqMap.get(team.getTeamNo()));
    }
}
```

## ⚠️ 마주친 문제점 & 해결

### **Problem 1: 동적 계산 필드로 정렬 불가**

**증상:**
```sql
SELECT ... 
WHERE ...
ORDER BY stage_choice.pass_yn  -- ← stage_choice는 조인해야 함
LIMIT 10 OFFSET 0

→ 트랙마다 스테이지가 다르면 LEFT JOIN 폭발 ❌
```

**해결:**
```java
// fullFetch = true
// DB: LIMIT 없이 전체 조회
// Service: 메모리에서 정렬 후 subList로 페이징

List<> all = queryMapper.selectAll(...);
all.sort(comparatorByComputedField);
List<> page = all.subList(from, to);
```

### **Problem 2: 팀원 정보 N+1 쿼리**

**증상:**
```
페이지당 20팀
→ 20팀 × (팀원 조회) = 20 쿼리! ❌
전체 로드 시 300팀 × 300쿼리 = 재앙 ❌
```

**해결:**
```java
// 현재 페이지 팀 번호만 모아서
List<Long> teamNos = teamList.stream()
    .map(::getTeamNo).toList();

// IN-query 1회
List<Members> all = mapper.selectByTeamNos(teamNos);

// 메모리 그룹화 + 주입
```

### **Problem 3: 페이지 경계에서 rowNo 꼬임**

**증상:**
```
페이지 2 (20~40행)를 조회할 때
rowNo가 1부터 20까지로 나옴 (실제로는 21~40이어야 함) ❌
```

**해결:**
```java
int startRow = (page - 1) * pageSize;
int rowNo = startRow + 1;  // ← 페이지 오프셋 고려

for (team : teamList) {
    team.setRowNo(rowNo++);
}
```

## 📊 성과

### Before (모든 요청을 fullFetch)

```
대시보드 열기 (일반 정렬)
├─ 전체 5000팀 로드
├─ 계산 필드 계산 (모두)
├─ 팀원 N+1 쿼리
└─ 응답: 3~5초 ❌
```

### After (하이브리드 경로)

```
일반 필드 정렬 (90% 케이스)
├─ DB LIMIT/OFFSET (페이지크기만)
├─ 팀원 IN-query 1회
└─ 응답: 200~300ms ✅

심사결과 정렬 (10% 케이스, 관리자)
├─ 전체 로드 (1회)
├─ 메모리 정렬
├─ 팀원 IN-query 1회
└─ 응답: 1~2초 ✅ (관리자 용도이므로 허용)
```

### 효과

```
✅ 일반 조회: 10배 속도 개선 (3초 → 300ms)
✅ 계산필드 필터링: 완전 지원
✅ 메모리: 최대 페이지크기 + 계산 필드 비용만
✅ 유연성: 요청에 따라 최적 경로 자동 선택
```

## 🎓 배운 점

### 1. 계산 필드와 페이징의 트레이드오프

> **"DB에 없는 필드로 정렬하면 전체 로드는 필수"**

- DB의 강점 = 대용량 정렬, 필터링
- Java의 강점 = 동적 계산, 유연한 로직
- 경계를 명확히 하자

### 2. 요청 분석 기반 경로 선택

> **"모든 요청이 동일한 경로를 취할 필요는 없다"**

```java
if (hasComputedField) {
    fullFetch();
} else {
    dbPaging();
}
```

→ 하이브리드 패턴은 상황에 따라 최적화!

### 3. N+1은 데이터 로드 후에도 발생

```
페이지네이션 후 팀원 조회
→ 서비스 레벨 페이징 덕에 쿼리 수는 줄었지만
→ 여전히 IN-query로 한 번에 처리해야 함
```

## 📌 핵심 코드 포인트

| 기법 | 코드 |
|------|------|
| 화이트리스트 | `allowedSortFields.contains(field)` |
| 경로 판정 | `isResultSort \|\| hasResultFilter` |

| nullsLast | `Comparator.nullsLast(...)` |
| IN-query | `selectByTeamNos(List<Long>)` |
| 페이지 오프셋 | `rowNo = (page-1)*size + 1` |
