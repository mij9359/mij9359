## 🎯 문제 정의

### 상황: 300개 팀을 언제 심사할 것인가?

```
📋 요구사항:
- 팀 수: 300개
- 운영 기간: 2026-07-09 ~ 2026-08-01 (선택된 요일만)
- 운영 시간: 10:00 ~ 18:00
- 팀당 발표시간: 8분
- 전환시간(팀 교체): 2분
- 휴식: 5팀마다 10분
- 점심: 11:40 ~ 13:00 (80분)
- 평가 정리: 30분 (옵션)

결과: 일정표 생성 (날짜·시간 배정)
```

### 복잡성

```
❌ 단순 계산: 10시간 × 60분 ÷ 10분(발표+전환) = 60팀/일
   → 300팀 ÷ 60 = 5일이면 된다?

✅ 실제: 5가지 제약이 얽혀 있음

1. 운영 요일 (토일요일 제외, 선택된 요일만)
2. 점심시간 (11:40 시작하면 13:00까지 모두 LUNCH)
3. 휴식 (5팀마다이지만, 점심 바로 전/후, 하루 끝 전에는 금지)
4. 발표시간 + 전환시간 (10분 슬롯)
5. 하루 경계 (12:18까지는 수용, 12:28은 다음 날로)

→ 이 5가지를 **모두 만족**시키는 일정을 만들어야 함!
```

### 기존 방식의 문제

```
❌ 수작업 엑셀:
- 운영팀이 300팀을 수동으로 배치
- 점심 시간이 겹치거나 휴식 배치가 중복됨
- 날짜가 바뀔 때마다 처음부터 다시 계산
- 오류 발생 확률 높음

❌ 단순 반복 로직:
for (int i = 0; i < teamCount; i++) {
    currentTime = currentTime.plusMinutes(10); // ← 점심 무시!
    if (currentTime.isAfter(dayEnd)) {
        currentDate = currentDate.plusDays(1); // ← 운영 요일 무시!
    }
}
```

## 💡 솔루션: 상태 머신 (State Machine) 기반 스케줄링

```
현재 상태: (날짜, 시간, 배치된팀수)

[루프] 매 반복마다 5가지 검사 (우선순위 순)
│
├─ (1) 운영 요일 아님?
│  └─ → 다음 날로 점프 (continue)
│
├─ (2) 점심시간 범위?
│  └─ → LUNCH 슬롯 생성, 점심 후로 이동 (continue)
│
├─ (3) 휴식 삽입 시점 & 조건?
│  └─ → BREAK 슬롯 생성, 휴식 후로 이동 (continue)
│
├─ (4) 발표시간이 하루 초과?
│  └─ → 다음 날로 점프 (continue)
│
└─ (5) 일반 발표 슬롯
   └─ → EVALUATION 슬롯 생성, 시간 진행

이 로직을 300번 반복하면 끝! ✅
```

## 🔥 핵심 구현 코드

```java
private List<EvaluationTimetableSlotVO> generateSlots(
        EvaluationScheduleDetailResVO detail, 
        int generationId, 
        int teamCount) {

    List<EvaluationTimetableSlotVO> slots = new ArrayList<>();
    
    LocalDate currentDate = LocalDate.parse(detail.getStartDate());
    LocalTime currentTime = LocalTime.parse(detail.getDayStart());
    LocalTime dayEnd      = LocalTime.parse(detail.getDayEnd());
    LocalTime lunchStart  = LocalTime.parse(detail.getLunchStart());
    LocalTime lunchEnd    = LocalTime.parse(detail.getLunchEnd());
    
    int slotOrder = 1;
    int lastBreakAfterSlotOrder = 0; // ← 점심/날짜변경 시 리셋하는 카운터
    
    // ===== 핵심: 상태 머신 반복 =====
    while (slotOrder <= teamCount) {
        
        // ─────────────────────────────────────
        // [Step 1] 운영 요일 검사
        // ─────────────────────────────────────
        if (!isAvailableDay(currentDate, detail)) {
            // 월/화/수만 운영하는데 목요일이면 스킵
            currentDate = currentDate.plusDays(1);
            currentTime = LocalTime.parse(detail.getDayStart());
            lastBreakAfterSlotOrder = slotOrder - 1; // 날짜 바뀌면 휴식 카운트 리셋
            continue; // ← 재평가!
        }
        
        // ─────────────────────────────────────
        // [Step 2] 점심시간 처리 (최우선)
        // ─────────────────────────────────────
        if (!currentTime.isBefore(lunchStart) && currentTime.isBefore(lunchEnd)) {
            // 11:40 ~ 13:00 범위에 있으면 LUNCH 슬롯
            
            slots.add(EvaluationTimetableSlotVO.builder()
                .slotType(SlotType.LUNCH)
                .evaluationDate(currentDate)
                .startTime(currentTime)
                .endTime(lunchEnd)
                .slotOrder(-1) // LUNCH는 순번 없음
                .build());
            
            lastBreakAfterSlotOrder = slotOrder - 1; // 점심 후 휴식 카운트 리셋
            currentTime = lunchEnd; // 13:00으로 이동
            continue; // ← 재평가!
        }
        
        // ─────────────────────────────────────
        // [Step 3] 휴식 삽입 (5팀마다)
        // ─────────────────────────────────────
        // 지금까지 발표슬롯 몇 개? slotOrder - lastBreakAfterSlotOrder - 1
        if ((slotOrder - lastBreakAfterSlotOrder - 1) 
            == detail.getBreakEveryTeamCount()) {
            
            LocalTime breakEnd = currentTime.plusMinutes(detail.getBreakMinutes());
            
            // 조건 1: 휴식이 점심과 겹치는가?
            boolean overlapsLunch = currentTime.isBefore(lunchStart) 
                                && breakEnd.isAfter(lunchStart);
            
            // 조건 2: 하루 종료 직전(10분 미만 남음)인가?
            boolean tooLateForBreak = !currentTime.isBefore(
                dayEnd.minusMinutes(detail.getBreakMinutes()));
            
            if (!overlapsLunch && !tooLateForBreak) {
                // 둘 다 OK → BREAK 슬롯 생성
                slots.add(EvaluationTimetableSlotVO.builder()
                    .slotType(SlotType.BREAK)
                    .evaluationDate(currentDate)
                    .startTime(currentTime)
                    .endTime(breakEnd)
                    .build());
                
                lastBreakAfterSlotOrder = slotOrder - 1; // 휴식 후 카운트 리셋
                currentTime = breakEnd;
                continue; // ← 재평가!
            }
            // 겹치거나 너무 늦으면 SKIP (continue 하지 않음 → 바로 Step 4로)
        }
        
        // ─────────────────────────────────────
        // [Step 4] 발표시간이 하루 초과?
        // ─────────────────────────────────────
        if (currentTime.plusMinutes(detail.getPresentationMinutes())
            .isAfter(dayEnd)) {
            
            // 12:28부터 8분 발표는 18:00을 넘음 → 다음 날로
            currentDate = currentDate.plusDays(1);
            currentTime = LocalTime.parse(detail.getDayStart());
            lastBreakAfterSlotOrder = slotOrder - 1; // 날짜 바뀌면 카운트 리셋
            continue; // ← 재평가!
        }
        
        // ─────────────────────────────────────
        // [Step 5] 일반 발표 슬롯 생성
        // ─────────────────────────────────────
        LocalTime slotEnd = currentTime.plusMinutes(detail.getPresentationMinutes());
        
        slots.add(EvaluationTimetableSlotVO.builder()
            .slotType(SlotType.EVALUATION)
            .evaluationDate(currentDate)
            .startTime(currentTime)
            .endTime(slotEnd)
            .slotOrder(slotOrder) // ← 팀 번호와 1:1 매핑
            .build());
        
        // 발표 + 전환시간 더하기
        currentTime = slotEnd.plusMinutes(detail.getTransitionMinutes());
        slotOrder++; // ← 팀 카운트 증가
        
        // continue 없음 → 루프 다시 시작 (Step 1부터 검사)
    }
    
    // ───────────────────────────────
    // [옵션] 마지막에 평가 정리 슬롯
    // ───────────────────────────────
    if (Boolean.TRUE.equals(detail.getSummaryEnabled())) {
        slots.add(EvaluationTimetableSlotVO.builder()
            .slotType(SlotType.SUMMARY)
            .evaluationDate(currentDate)
            .startTime(currentTime)
            .endTime(currentTime.plusMinutes(detail.getSummaryMinutes()))
            .build());
    }
    
    return slots;
}

// ──────────────────────────────────────────
// Helper: 운영 요일 검사
// ──────────────────────────────────────────
private boolean isAvailableDay(LocalDate date, EvaluationScheduleDetailResVO detail) {
    DayOfWeek dow = date.getDayOfWeek();
    
    if (dow == DayOfWeek.MONDAY)    return detail.getMon();
    if (dow == DayOfWeek.TUESDAY)   return detail.getTue();
    if (dow == DayOfWeek.WEDNESDAY) return detail.getWed();
    if (dow == DayOfWeek.THURSDAY)  return detail.getThu();
    if (dow == DayOfWeek.FRIDAY)    return detail.getFri();
    if (dow == DayOfWeek.SATURDAY)  return detail.getSat();
    if (dow == DayOfWeek.SUNDAY)    return detail.getSun();
    
    return false;
}
```

## ⚠️ 마주친 문제점 & 해결

### **Problem 1: 휴식이 점심과 겹침**

**증상:**
```
11:30 휴식 시작 → 11:40 끝나야 하는데
점심이 11:40 시작 → "11:30-11:40"은 점심인가 휴식인가? ❌

또는 13:00 점심 끝 → 13:10 휴식 시작?
그럼 13:00-13:10이 "공백"이 됨 ❌
```

**해결:**
```java
// 휴식이 점심과 겹치는지 사전 검사
boolean overlapsLunch = currentTime.isBefore(lunchStart) 
                    && breakEnd.isAfter(lunchStart);

if (overlapsLunch) {
    // 휴식 생략 → Step 5로 진행 (발표 슬롯 생성)
}
```

### **Problem 2: 하루 종료 직전에 휴식 삽입**

**증상:**
```
17:58 휴식 시작 → 17:58 + 10분 = 18:08 (18:00 초과!) ❌
또는 18:00 정각에 휴식이 끝나면
그 다음 발표 슬롯을 생성할 수 없음 ❌
```

**해결:**
```java
// 하루 종료까지 휴식 시간이 남는가?
boolean tooLateForBreak = !currentTime.isBefore(
    dayEnd.minusMinutes(detail.getBreakMinutes()));

if (tooLateForBreak) {
    // 휴식 생략 → 그냥 발표 슬롯으로 (오버플로우되면 Step 4에서 처리)
}
```

### **Problem 3: 날짜 경계에서 카운트 꼬임**

**증상:**
```
2026-07-09 마지막 슬롯 (5팀 발표 후 휴식 삽입)
→ 2026-07-10으로 바뀜
→ 다음 슬롯이 6팀인데, 휴식 카운트가 "1, 2, 3, 4, 5"가 아니라
  "이전 날짜의 5+1" 상태로 남음 ❌
```

**해결:**
```java
// 날짜/점심/휴식 직후마다 카운트 리셋
if (!isAvailableDay(...)) {
    lastBreakAfterSlotOrder = slotOrder - 1; // ← 리셋
}

if (점심) {
    lastBreakAfterSlotOrder = slotOrder - 1; // ← 리셋
}

if (휴식생성) {
    lastBreakAfterSlotOrder = slotOrder - 1; // ← 리셋
}
```

### **Problem 4: 단순 while 루프로는 5가지 제약을 동시에 못 만족**

**증상:**
```
for (int i = 0; i < 300; i++) {
    if (점심) LUNCH 생성;
    else if (휴식) BREAK 생성;
    else EVALUATION 생성;
    
    currentTime += 10분;
}

→ 이 방식은 한 루프에서 "한 가지만" 처리
   점심을 생성하면 시간을 13:00으로 점프하고
   다음 루프에서 "휴식 검사"를 하는데
   이미 점심 이후 상태이므로 조건이 꼬임 ❌
```

**해결:**
```java
while (slotOrder <= teamCount) {
    // Step 1: 요일 검사 → continue (재평가)
    // Step 2: 점심 검사 → continue (재평가)
    // Step 3: 휴식 검사 → continue (재평가)
    // Step 4: 오버플로우 검사 → continue (재평가)
    // Step 5: 발표 슬롯 생성 → continue 없음 (루프 진행)
}

// → 매 루프마다 "현재 상태"를 Step 1부터 재평가
// → 상태가 변하면 (점심으로 점프, 날짜 변경) continue로 처음부터
// → 5가지 제약을 모두 만족!
```

## 🎓 배운 점

### 1. 상태 머신의 강력함

> **"여러 제약을 동시에 만족시키려면 상태를 재평가하라"**

- 간단한 반복문으로는 불가능
- **매 iteration마다 "현재 상태"를 우선순위 순으로 검사**
- 상태가 변하면 (점프, 리셋) `continue`로 처음부터

### 2. 포인터 변수의 중요성

- `lastBreakAfterSlotOrder`: "지난 휴식 이후 몇 팀을 배치했는가?"
- 이 포인터를 **날짜/점심/휴식 직후마다 리셋**하지 않으면 카운트 꼬임
- **상태 변화점마다 명시적 리셋** 필수

### 3. 엣지 케이스의 폭발

실제 운영 중 발견된 버그:
- 점심 정확히 시작 시점 (11:40)에 휴식 끝남
- 마지막 발표가 17:52 시작 (18:00 안에 끝남)
- 토요일 운영 요일에 포함될 때

→ **단위 테스트 + 통합 테스트**로 충분하지 않음
→ **실제 운영 데이터로 재검증** 필수

### 4. 파라미터 드리븐 설계의 힘

- 알고리즘 로직은 고정
- 모든 조건(점심시간, 휴식 주기, 요일)은 파라미터로 입력
- → DB 데이터 수정만으로 시간표 재생성 가능

```
while (슬롯 생성 < 300개) {
    ├─ Step 1: isAvailableDay? ──→ 다음 날로 (continue)
    ├─ Step 2: 점심 범위?      ──→ LUNCH 슬롯 (continue)
    ├─ Step 3: 휴식 시점?      ──→ BREAK 슬롯 (continue, 조건부)
    ├─ Step 4: 발표 초과?      ──→ 다음 날로 (continue)
    └─ Step 5: 발표 슬롯       ──→ 시간 진행, slotOrder++
}
```

→ 한 번의 `while` 루프로 **5가지 제약을 모두 만족**하는 시간표 자동 생성! ✅
