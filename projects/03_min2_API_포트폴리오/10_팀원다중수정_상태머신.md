# 👥 팀원 다중 수정 API: 식별자 변경 분기 & 가등록 상태머신

## 🎯 문제 정의

### 상황: 팀원 정보 수정

```
팀원 수정 요청:

[Before]
├─ 팀원명: 김철수
├─ 연락처: 010-1234-5678
├─ 이메일: kim@email.com
└─ state: 0 (정상)

[After]
├─ 팀원명: 김철수 (동일)
├─ 연락처: 010-9999-9999 (변경)
├─ 이메일: kim.new@email.com (변경)
└─ state: ? (0 유지 vs 3 변경?)
```

### 복잡성

```
❌ 문제 1: 같은 사람 vs 다른 사람 판별 어려움

Case A: 연락처만 수정
├─ 010-1234-5678 → 010-9999-9999
├─ 같은 사람이지만 연락처 변경
└─ state는 0 유지해야 함 ✅

Case B: 다른 사람으로 교체
├─ kim@email.com → lee@email.com
├─ 전혀 다른 사람
└─ state를 3 (가등록)으로 바꿔야 함
   (이메일 인증 전까지는 정상 회원이 아님)

어떻게 구분할 것인가? ❌

❌ 문제 2: 이력형 수정의 복잡성

일반 UPDATE:
UPDATE team_member SET ... WHERE tm_no = ?

이력형 수정:
1. 기존 행 state = 1 (논리적 삭제)
2. 새 행 INSERT (같은 정보 복사 + 필드 변경)
3. tm_jang (팀장/팀원 구분) 승계?

❌ 문제 3: 팀장 정보 승계

기존:
├─ tm_jang = 2 (팀장)
└─ 이전까지의 팀원 시퀀스 번호

수정 후에도:
└─ 새 행의 tm_jang = 2 (동일하게!)

이 값을 어떻게 알아낼 것인가? (SELECT BEFORE INSERT)
```

## 💡 솔루션: 도메인 메서드 + 2-Step 쿼리

```
[Step 1] 기존 상태 스냅샷
└─ originalState = 0 또는 3
└─ originalUserNo = 기존 유저

[Step 2] 식별자 변경 판별
├─ 기존 유저와 새 유저가 같은가?
│  ├─ YES (같은 사람)
│  │  └─ retainOriginalState(originalState)
│  │     → state 유지 (0이면 0, 3이면 3)
│  │
│  └─ NO (다른 사람)
│     └─ changeToPendingMember(newUserNo)
│        → state = 3 (가등록)
│        → 이메일 인증 토큰 발송

[Step 3] 이력형 수정 실행
├─ 기존 행 state = 1 (마킹)
├─ 새 행 INSERT (상태 적용)
└─ tm_jang은 <selectKey>로 자동 승계
```

## 🔥 핵심 구현 코드

### Step 1: Service 진입점

```java
@Transactional
public void modifyTeamMemberBulk(ModifyTeamMemberBulkDto request) {
    
    // ──────────────────────────────────────────────────────
    // [Step 1] 기존 상태 스냅샷
    // ──────────────────────────────────────────────────────
    
    // 기존 팀원의 state를 먼저 캡처
    int originalState = applyQueryMapper.getTeamMemberState(request.getTmNo())
        .orElse(3);
    
    // 기존 팀원의 user_no를 먼저 캡처
    Long originalUserNo = applyQueryMapper.getTeamMemberUserNo(request.getTmNo())
        .orElseThrow(() -> new OperationErrorException(
            ErrorCode.FAIL_TO_MODIFY_TEAM_MEMBER,
            "팀원을 찾을 수 없습니다"));
    
    log.info("[Member Modify] tm_no={}, originalState={}, originalUserNo={}",
             request.getTmNo(), originalState, originalUserNo);
    
    // ──────────────────────────────────────────────────────
    // [Step 2] 기존 행을 논리적으로 삭제 (state=1)
    // ──────────────────────────────────────────────────────
    
    if (applyCommandMapper.updateTeamMemberState(
            request.getTmNo(), request.getUserNo(), 1) < 1) {
        throw new OperationErrorException(
            ErrorCode.FAIL_TO_MODIFY_TEAM_MEMBER,
            "기존 팀원 정보를 처리하는 중 오류 발생");
    }
    
    // ──────────────────────────────────────────────────────
    // [Step 3] 식별자 변경 여부에 따라 상태 결정
    //         (이 부분이 핵심!)
    // ──────────────────────────────────────────────────────
    
    resolveMemberIdentityAndState(request, originalUserNo, originalState);
    
    // ──────────────────────────────────────────────────────
    // [Step 4] 새 행 INSERT (tm_jang은 자동 승계)
    // ──────────────────────────────────────────────────────
    
    if (applyCommandMapper.modifyTeamMemberBulk(request) < 1) {
        throw new OperationErrorException(
            ErrorCode.FAIL_TO_MODIFY_TEAM_MEMBER,
            "팀원 정보 업데이트 실패");
    }
    
    log.info("[Member Modify] tm_no={} 수정 완료 (new state={})",
             request.getTmNo(), request.getState());
}
```

### Step 2: 핵심 식별자 판별 로직

```java
private void resolveMemberIdentityAndState(ModifyTeamMemberBulkDto request,
                                          Long originalUserNo, 
                                          int originalState) {
    
    // ──────────────────────────────────────────────────────
    // 새로운 연락처/이메일로 매칭되는 user_no 찾기
    // ──────────────────────────────────────────────────────
    
    Long findUserNo = userQueryMapper
        .findUserNoToPhoneAndEmail(request.getTmPhone(), request.getTmEmail())
        .orElseThrow(() -> new OperationErrorException(
            ErrorCode.NOT_FOUND_USERS_PHONE,
            "입력한 연락처/이메일에 매칭되는 사용자가 없습니다"));
    
    // ──────────────────────────────────────────────────────
    // Case A: 같은 사람인가?
    // ──────────────────────────────────────────────────────
    
    if (Objects.equals(originalUserNo, findUserNo)) {
        
        log.info("[Member Identity] 같은 사람 (user_no={}) → state 유지 ({})",
                 findUserNo, originalState);
        
        // 도메인 메서드: state를 기존 값으로 유지
        request.retainOriginalState(originalState);
        
        // DTO 내부:
        // public void retainOriginalState(int state) {
        //     this.state = state;  // 0 유지 또는 3 유지
        // }
        
    } else {
        
        log.info("[Member Identity] 다른 사람 ({}→{}) → state=3 (가등록)",
                 originalUserNo, findUserNo);
        
        // ──────────────────────────────────────────────────────
        // Case B: 다른 사람 → 중복 신청 검증 + 가등록 처리
        // ──────────────────────────────────────────────────────
        
        // 이 사람이 이미 같은 트랙에 신청했는가?
        applyQueryService.checkTeamMemberApply(
            request.getTrackNo(), 
            request.getTmPhone());
        
        // 도메인 메서드: 가등록 상태로 전환 + user_no 설정
        request.changeToPendingMember(findUserNo);
        
        // DTO 내부:
        // public void changeToPendingMember(Long userNo) {
        //     this.tmUserNo = userNo;
        //     this.state = 3;  // ← 가등록!
        // }
        
        // 이메일 인증 토큰 발송 준비
        sendVerificationEmail(findUserNo, request.getTmEmail());
    }
}
```

### Step 3: 도메인 메서드 (DTO)

```java
public class ModifyTeamMemberBulkDto {
    
    private Long tmNo;           // 팀원 ID (PK)
    private Long trackNo;        // 트랙 ID
    private Long teamNo;         // 팀 ID
    private Long userNo;         // 현재 로그인 사용자
    private Long tmUserNo;       // 팀원의 user_no (변경될 수 있음)
    private String tmPhone;      // 팀원 연락처
    private String tmEmail;      // 팀원 이메일
    private int state;           // 팀원 상태 (0=정상, 3=가등록)
    
    // ──────────────────────────────────────────────────────
    // [도메인 메서드] 같은 사람 → state 유지
    // ──────────────────────────────────────────────────────
    
    public void retainOriginalState(int originalState) {
        this.state = originalState;
        // 로직: state를 변경하지 않음 (0이면 0, 3이면 3 유지)
    }
    
    // ──────────────────────────────────────────────────────
    // [도메인 메서드] 다른 사람 → 가등록 전환
    // ──────────────────────────────────────────────────────
    
    public void changeToPendingMember(Long newUserNo) {
        this.tmUserNo = newUserNo;    // 새 user_no 설정
        this.state = 3;               // ← 가등록!
        // 로직: 이메일 인증 토큰 필요 (별도 처리)
    }
}
```

### Step 4: MyBatis - tm_jang 자동 승계

```xml
<!-- ApplyCommandMapper.xml:1021 -->
<update id="updateTeamMemberState">
    UPDATE team_member
    SET state = #{state}
    WHERE tm_no = #{tmNo}
</update>

<!-- 핵심: <selectKey order="BEFORE"> 로 INSERT 전 tm_jang 조회 -->
<insert id="modifyTeamMemberBulk">
    <selectKey resultType="Integer" order="BEFORE" keyProperty="tmJang">
        <!-- 기존 팀원의 tm_jang (팀장 순번)을 먼저 가져옴 -->
        SELECT COALESCE(MAX(tm_jang), 1)
        FROM team_member
        WHERE track_no = #{trackNo} 
          AND team_no = #{teamNo} 
          AND user_no = #{tmUserNo}
        LIMIT 1
    </selectKey>
    
    INSERT INTO team_member 
        (track_no, team_no, user_no, tm_phone, tm_email, tm_jang, state, created_at)
    VALUES
        (#{trackNo}, #{teamNo}, #{tmUserNo}, #{tmPhone}, #{tmEmail}, 
         #{tmJang}, #{state}, NOW())
</insert>
```

## ⚠️ 마주친 문제점 & 해결

### **Problem 1: state 변경 잘못**

**증상:**
```java
// 나쁜 예
if (sameUser) {
    request.setState(0);  // ← 항상 0으로? 틀렸다!
} else {
    request.setState(3);
}

// 기존이 3(가등록)이었으면 3 유지해야 하는데
// 0으로 바뀌어서 인증된 상태가 됨 ❌
```

**해결:**
```java
// 도메인 메서드로 명시적 처리
request.retainOriginalState(originalState);  // ← 기존 값 그대로
request.changeToPendingMember(newUserNo);    // ← 새 가등록 상태
```

### **Problem 2: 중복 신청 검증 누락**

**증상:**
```
팀원을 "이미 신청한 다른 사람"으로 교체
→ 중복 신청 체크 없이 INSERT
→ 데이터 불일치 ❌
```

**해결:**
```java
if (isDifferentPerson) {
    // 다른 사람으로 교체할 때만 검증
    applyQueryService.checkTeamMemberApply(trackNo, tmPhone);
    // ← 이미 신청했으면 예외 발생
}
```

### **Problem 3: tm_jang 값 누락**

**증상:**
```
새 행 INSERT 시 tm_jang을 명시하지 않음
→ NULL 또는 0으로 기본값 설정 ❌
→ 팀장 구분 불가
```

**해결:**
```xml
<!-- <selectKey order="BEFORE">로 INSERT 전 기존 값 가져옴 -->
<selectKey ... keyProperty="tmJang">
    SELECT MAX(tm_jang) FROM team_member WHERE ...
</selectKey>
```

## 📊 성과

### Before (단순 UPDATE)

```
기존 코드:
UPDATE team_member 
SET tm_phone = ?, tm_email = ?, state = 0  
WHERE tm_no = ?

문제:
├─ state를 항상 0으로 → 가등록 무시 ❌
├─ tm_jang 손실 ❌
└─ 다른 사람으로 교체해도 검증 없음 ❌
```

### After (이력형 + 상태머신)

```
새 코드:
1. 기존 상태 캡처
2. 기존 행 논리 삭제
3. 식별자 판별 → 상태 결정
4. 새 행 INSERT (tm_jang 승계)

성과:
├─ 상태 보존: Case A ✅
├─ 가등록 전환: Case B ✅
├─ 중복 신청 방지 ✅
└─ 팀장 정보 유지 ✅
```

## 🎓 배운 점

### 1. 상태머신은 도메인 메서드로 캡슐화

> **"Setter 남용하지 말고, 상태 전이는 메서드로"**

```
❌ request.setState(newValue);

✅ request.retainOriginalState(original);
   request.changeToPendingMember(userId);
```

→ 의도가 명확, 유지보수 쉬움

### 2. 스냅샷 먼저 캡처하기

> **"이전 값이 필요하면 맨 처음에 잡아둬라"**

```
기존 상태를 먼저 캡처
→ 로직 진행
→ 상태 결정 시 원본과 비교
```

### 3. 식별자 변경은 비즈니스 로직

> **"다른 사람이면 다른 처리를 해야 한다"**

```
같은 사람: state 유지 (이미 인증됨)
다른 사람: state = 3 (재인증 필요)
```
