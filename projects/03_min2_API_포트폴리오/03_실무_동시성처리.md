## 🎯 문제 정의

### 상황 1: 배정 중 데이터 불일치

```
시나리오:
┌─────────────────────────────────────────────────┐
│ 관리자 A: "10개 팀을 다른 트랙으로 배정" 클릭  │
│   UPDATE team SET assigned_track_no = 2         │
│   WHERE team_no IN (123, 124, 125, ...)         │
└─────────────────────────────────────────────────┘
         ↓ (같은 순간)
┌─────────────────────────────────────────────────┐
│ 관리자 B: "팀 125의 상태를 탈락으로 변경" 클릭 │
│   UPDATE team SET pt_pass = 0                   │
│   WHERE team_no = 125                           │
└─────────────────────────────────────────────────┘

결과:
UPDATE WHERE team_no IN (...)의 조건에 pt_pass = 1만 있으면
팀 125는 UPDATE 되지 않음 ❌

UI 화면: "10개 팀 배정 완료!" ✅
DB 상태: 9개만 배정됨 ❌
운영팀: "어? 10개라고 했는데 9개네?"
```

### 상황 2: 0점/면제팀이 보정계수를 왜곡

```
실제 운영 데이터:

[상황]
- 점수가 입력되지 않은 팀 (0점)
- 평가 면제팀

[문제]
이 팀들이 보정계수 계산에 포함되면
→ overallRawAvg가 낮아짐
→ 다른 팀의 보정계수가 비정상적으로 커짐 ❌

예시:
원래 평균: 62.35
0점팀 1개 추가: (62.35*99 + 0) / 100 = 61.73
→ 서울 권역 보정계수: 61.73 / 55 = 1.122 (원래 1.134였는데 낮아짐)
→ 다른 팀들이 손해 봄 ❌
```

╚─ 요청 수 (10) != 실제 업데이트 수 (9)
   └─ 예외 발생
   └─ @Transactional으로 전체 롤백 ✅

### 솔루션 1: 트랜잭션 무결성

```java
// ═════════════════════════════════════════════════════════
// [API 1] 팀을 다른 트랙으로 배정
// ═════════════════════════════════════════════════════════

@Transactional  // ← 핵심: 트랜잭션 보장
public void updateAssignTeamsToTrack(AssignTeamsToTrackReqVO reqVO) {
    int requestedCount = reqVO.getTeamNoList().size();
    
    // MyBatis UPDATE 실행
    int updatedCount = pointStageCommandMapper.updateTeamAssignedTrack(reqVO);
    
    // 요청 개수 != 실제 업데이트 개수 → 예외 발생 → 롤백
    if (requestedCount != updatedCount) {
        throw new OperationErrorException(
            ErrorCode.ASSIGNED_TRACK_COUNT_MISMATCH,
            String.format("요청 팀 수(%d) != 실제 업데이트(%d)",
                         requestedCount, updatedCount));
    }
    
    // 통과하면 정상 커밋
}

// ═════════════════════════════════════════════════════════
// [API 2] 배정 취소
// ═════════════════════════════════════════════════════════

@Transactional
public void updateCancelAssignTeamsToTrack(CancelAssignTeamsReqVO reqVO) {
    int requestedCount = reqVO.getTeamNoList().size();
    
    // assigned_track_no를 NULL로 설정
    int updatedCount = pointStageCommandMapper.updateTeamCancelAssignedTrack(reqVO);
    
    if (requestedCount != updatedCount) {
        throw new OperationErrorException(
            ErrorCode.CANCEL_ASSIGN_TRACK_COUNT_MISMATCH);
    }
}

// ═════════════════════════════════════════════════════════
// [MyBatis XML] UPDATE 쿼리
// ═════════════════════════════════════════════════════════

<update id="updateTeamAssignedTrack">
    UPDATE team
    SET assigned_track_no = #{assignedTrackNo},
        updated_at = NOW(),
        updated_by = #{updatedBy}
    WHERE team_no IN
    <foreach collection="teamNoList" item="teamNo" 
             open="(" separator="," close=")">
        #{teamNo}
    </foreach>
    AND state = 0                           <!-- 삭제되지 않은 팀만 -->
    AND assigned_track_no IS NULL           <!-- 아직 배정 안 된 팀만 -->
    AND pt_pass = 1                         <!-- 합격 팀만 -->
    AND team_submission_exclusion = 0       <!-- 면제 팀 제외 -->
</update>
```

**동작 흐름:**

```
요청: [팀123, 팀124, 팀125, ...]
║
╠─ 데이터 검증 (WHERE 조건 확인)
║  ├─ state = 0? (삭제 안 됨)
║  ├─ assigned_track_no IS NULL? (아직 배정 안 됨)
║  ├─ pt_pass = 1? (합격 팀)
║  └─ team_submission_exclusion = 0? (면제 아님)
║
╠─ UPDATE 실행
║  └─ 실제 영향받은 행 수 반환
║
╚─ 검증
   ├─ 요청 수 == 실제 UPDATE 수?
   ├─ YES → 커밋 ✅
   └─ NO → 예외 발생 → 롤백 ❌
```

### 솔루션 2: 0점/면제팀 확장 (기존 보정 로직에 통합)

```java
private void applyScoreCorrectionAndRanking(
        List<AssignedPointListResVO> teams, 
        String sortType, 
        String sortOrder) {

    // ───────────────────────────────────────────
    // [핵심] 순수 원점수 추출
    // ───────────────────────────────────────────
    Function<AssignedPointListResVO, Double> getPureScore = t -> {
        String s = t.getScoreAvg();
        
        // 0점/면제팀 여부 판별
        if (s == null || s.isEmpty() || "평가 면제".equals(s)) {
            return 0.0; // ← 0점팀 반환
        }
        
        try {
            double scoreAvg = Double.parseDouble(s);
            double bonus = (t.getPtAddition() != null) 
                         ? t.getPtAddition() : 0.0;
            double penalty = (t.getPtAdditionMinus() != null) 
                          ? t.getPtAdditionMinus() : 0.0;
            return scoreAvg - bonus + penalty; // 순수 원점수
        } catch (NumberFormatException e) {
            return 0.0;
        }
    };

    // ───────────────────────────────────────────
    // [핵심] 통계 모수 분리
    // 0점/면제팀 제외, 점수 > 0인 팀만
    // ───────────────────────────────────────────
    List<AssignedPointListResVO> scoredTeams = teams.stream()
        .filter(t -> getPureScore.apply(t) > 0)
        .collect(Collectors.toList());

    // 평가 데이터가 없으면 조기 반환
    if (scoredTeams.isEmpty()) {
        // 0점 팀만 남음 → 정렬하지 않고 반환
        return;
    }

    // ───────────────────────────────────────────
    // [Step 3~5] 보정계수 & Z-Score 계산
    // (이전과 동일, scoredTeams로만 계산)
    // ───────────────────────────────────────────
    
    double overallRawAvg = scoredTeams.stream()
        .mapToDouble(getPureScore::apply)
        .average().orElse(0.0);

    Map<Long, Double> regionalAvgMap = scoredTeams.stream()
        .filter(t -> t.getPgNo() != null)
        .collect(Collectors.groupingBy(
            AssignedPointListResVO::getPgNo,
            Collectors.averagingDouble(getPureScore::apply)
        ));

    // 보정계수 & 보정원점수 (점수 > 0인 팀만)
    for (AssignedPointListResVO team : teams) {
        double pureScore = getPureScore.apply(team);
        
        if (pureScore > 0) {
            double regAvg = regionalAvgMap
                .getOrDefault(team.getPgNo(), overallRawAvg);
            double factor = (regAvg <= 0) ? 1.0 
                          : overallRawAvg / regAvg;
            
            team.setCorrectionFactor(roundToFour(factor));
            team.setAdjRawScore(roundToFour(pureScore * factor));
        } else {
            // 0점/면제팀 기본값
            team.setCorrectionFactor(1.0);
            team.setAdjRawScore(0.0);
            team.setFinalZScore(0.0); // ← Z점수 0.0 고정
        }
    }

    // ───────────────────────────────────────────
    // [Step 7~9] 정렬 & 등수
    // ───────────────────────────────────────────
    
    // 면제팀 최상단, 그 다음 점수순, 0점팀 최하단
    Predicate<AssignedPointListResVO> isExempt = t ->
        "평가 면제".equals(t.getScoreAvg())
        || (t.getTeamSubmissionExclusion() != null 
            && t.getTeamSubmissionExclusion() == 1);

    teams.sort(Comparator
        .comparing(isExempt::test, Comparator.reverseOrder())
        .thenComparing(AssignedPointListResVO::getResultScore, 
                      Comparator.reverseOrder())
        .thenComparing(AssignedPointListResVO::getFinalZScore, 
                      Comparator.reverseOrder()));

    // 등수 부여
    int i = 0, currentRankBase = 1;
    while (i < teams.size()) {
        int j = i;
        while (j < teams.size() && isSameGroup(teams.get(i), teams.get(j))) {
            j++;
        }
        int groupSize = j - i;
        int rank = currentRankBase + groupSize - 1;
        for (int k = i; k < j; k++) {
            teams.get(k).setFinalRank(rank);
        }
        i = j;
        currentRankBase += groupSize;
    }
}
```

**결과 (0점팀 처리 전후):**

```
[Before] 0점팀이 평균에 포함
─────────────────────────────────
전체 평균: 60.50 (낮음)
서울 권역: 평균 55, 보정계수 = 60.50/55 = 1.100

[After] 0점팀 제외
─────────────────────────────────
전체 평균: 62.35 (정상)
서울 권역: 평균 55, 보정계수 = 62.35/55 = 1.134

차이: 보정계수가 1.100 → 1.134로 증가
→ 서울 팀들이 더 공정하게 평가됨 ✅
```

## ⚠️ 마주친 문제점

### **Problem 1: 요청 수 != 업데이트 수 (초기 배포)**

**증상:**
```
관리자: "10개 팀 배정 완료!"
DB 상태: 9개만 배정됨

원인:
WHERE 조건에 pt_pass = 1이 없음
→ 합격/탈락 상태 무관하게 모두 배정 시도
→ 나중에 탈락팀으로 표시된 팀은 조건에 맞지 않아 스킵
```

**해결:**
```sql
AND pt_pass = 1  -- ← 추가
AND team_submission_exclusion = 0  -- ← 추가
```

### **Problem 2: 첫 배포 후 0점팀 문제 발견**

**증상:**
```
보정계수가 갑자기 비정상적으로 작아짐
(1.234에서 1.089로 떨어짐)

원인:
실제 운영 데이터에 "점수 입력 안 된 팀" 존재
→ scoreAvg = NULL 또는 "평가 면제"
→ 이 팀들이 통계에 포함되어 평균 왜곡
```

**해결:**
```java
// 통계 모수 필터링
List<AssignedPointListResVO> scoredTeams = teams.stream()
    .filter(t -> getPureScore.apply(t) > 0)
    .collect(toList());
```

### **Problem 3: 0점팀의 등수 처리**

**증상:**
```
0점팀을 어디에 배치할 것인가?

옵션 1: 등수 최하단
→ 탈락 팀과 구별 안 됨 ❌

옵션 2: 등수 부여 안 함
→ 최종 점수표에 NULL
→ 데이터 불일치 ❌

옵션 3: Z점수 0.0 고정, 등수는 계산
→ 모두 동점 처리 (999등까지 가능)
→ 데이터 일관성 유지 ✅
```

**해결:**
```java
// 0점팀 Z점수 0.0 고정
if (getPureScore.apply(team) > 0) {
    double zScore = (dispScore - adjAvg) / adjStd;
    team.setFinalZScore(roundToFour(zScore));
} else {
    team.setFinalZScore(0.0); // ← 0.0 고정
}
```

## 🎓 배운 점

### 1. 트랜잭션의 중요성

> **"부분 성공은 실패보다 나쁘다"**

- UPDATE가 "모두 성공" 또는 "모두 실패"해야 함
- 부분 업데이트는 데이터 불일치를 만듦
- **요청 수 == 업데이트 수 검증**이 필수

### 2. 실제 운영 데이터의 예측 불가능성

> **"개발 환경에서는 보이지 않던 버그가 운영에서 터진다"**

실제 문제들:
- 0점 팀 (평가 안 함)
- 평가 면제 팀
- 상태가 중간에 바뀐 팀
- 날짜/시간대별 차이

→ **단위 테스트 + 통합 테스트 + 실제 데이터 시뮬레이션 필수**

### 3. 필터링의 강력함

> **"한 번의 필터로 여러 엣지 케이스를 처리하라"**

```java
.filter(t -> getPureScore.apply(t) > 0)
```

이 한 줄이:
- 0점 팀 제외
- 면제 팀 제외
- 데이터 오류 팀 제외
→ 모두 해결!
