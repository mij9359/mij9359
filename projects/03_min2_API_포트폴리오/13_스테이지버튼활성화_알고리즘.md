## 🎯 문제 정의

### 상황: 트랙맵 스테이지 버튼

```
트랙 "도전기업":

[그룹 1]
├─ 공지 (sbt_no=3)                    활성화? 누구나 ✅
├─ 신청 (sbt_no=5)  ← Stage Order=1   활성화? 로그인하고 미신청 ⚠️
├─ 자료제출 (sbt_no=9)                활성화? 신청했고 기간 내 ⚠️

[그룹 2] ← 그룹1의 신청 이후
├─ 심사공지 (sbt_no=3)                활성화? 신청했으면 ⚠️
├─ 심사결과 (sbt_no=5)                활성화? 심사 후 ⚠️

[그룹 3] ← 그룹2의 심사 합격 후
├─ 최종공지
├─ 최종신청

요구사항:
├─ 비로그인: 공지(3) + 신청(5)까지만 보여줘
├─ 로그인 미신청: 신청(5) 버튼 활성화 조건 세분화
├─ 신청했으면: 다음 그룹의 버튼들 활성화
├─ 심사에서 합격: 그 다음 그룹 활성화
├─ 심사에서 탈락: 더 이상 버튼 비활성화
```

### 복잡성

```
❌ 문제 1: 버튼 활성화 조건이 다양함

신청 버튼:
├─ 비로그인: 활성화 (클릭 → 로그인)
├─ 로그인 + 미신청: 활성화
├─ 이미 신청: 비활성화
├─ 탈락: 비활성화
├─ 다른 트랙 신청 중: 비활성화? 활성화? ⚠️

버튼마다 이런 조건들이 있음
→ 분기가 수십~수백 가지

❌ 문제 2: 단계별 잠금/해제

Step 1: 신청 전 → 신청 버튼만 활성화
Step 2: 신청 후 → 자료제출 활성화, 신청 비활성화
Step 3: 자료제출 후 → 심사 대기
Step 4: 심사 합격 → 최종 단계 버튼 활성화
Step 5: 심사 탈락 → 모든 버튼 비활성화

이 흐름을 놓치면 사용자가 잘못된 버튼 클릭 가능 ❌

❌ 문제 3: 심사 스테이지 여부에 따른 분기

어떤 그룹은 심사가 있음:
└─ 순서: 신청 → 자료제출 → 심사 → 최종

어떤 그룹은 심사가 없음:
└─ 순서: 신청 → 자료제출 → 최종 (심사 생략!)

같은 로직으로 처리 불가 ❌
```

## 💡 솔루션: currentOrder 추상화 + 의사결정 트리

```
핵심 아이디어: 
"사용자가 접근 가능한 스테이지 순서의 한계선을 currentOrder로 표현"

currentOrder = Integer.MAX_VALUE  (모든 버튼 활성화)
         또는 = 특정 order 값    (그때까지만 활성화)
         또는 = 0                (모든 버튼 비활성화)

예:
├─ currentOrder = 5 → Order 1~5 버튼만 활성화
├─ currentOrder = MAX → 모든 버튼 활성화 (관리자)
└─ currentOrder = 0 → 모든 버튼 비활성화 (탈락자)

의사결정 트리:
├─ [비로그인?] → currentOrder = apply_order (신청 단계까지)
├─ [로그인 + 신청함?]
│  ├─ [심사 있음?]
│  │  └─ [합격?] → currentOrder = next_step_order
│  │     [탈락?] → currentOrder = 0
│  │     [미심사?] → currentOrder = apply_order
│  └─ [심사 없음?] → currentOrder = MAX
└─ [로그인 + 미신청?] → currentOrder = apply_order
```

## 🔥 핵심 구현 코드

```java
public ArrayList<StageListInfoVO> getStageList(
        Long userNo, 
        Long trackNo, 
        Long sgNo) {
    
    // ──────────────────────────────────────────────────────
    // [데이터 로드]
    // ──────────────────────────────────────────────────────
    
    boolean hasAppliedAsMember = mapQueryMapper
        .hasAppliedAsMember(trackNo, userNo);
    
    ArrayList<StageListInfoVO> stageList = 
        mapQueryMapper.getStageList(trackNo, sgNo);
    
    Integer applyOrder = mapQueryMapper
        .getNotLoginApplyStageOrder(trackNo, sgNo);
    
    if (applyOrder == null) applyOrder = Integer.MAX_VALUE;

    // ──────────────────────────────────────────────────────
    // [의사결정 1] 비로그인 경로
    // ──────────────────────────────────────────────────────
    
    if (userNo == null) {
        log.info("[Stage List] 비로그인 경로: trackNo={}, sgNo={}", 
                 trackNo, sgNo);
        
        Integer cnt = mapQueryMapper.applyStageCount(trackNo, sgNo);
        if (cnt < 1) applyOrder = 0;  // 신청 스테이지 없음 → 모두 비활성
        
        for (StageListInfoVO stage : stageList) {
            // 비로그인용 버튼만 조회
            stage.setStageButtonList(
                mapQueryMapper.getNotLoginButtonList(stage.getStageNo(), applyOrder));
        }
        
        return stageList;
    }

    // ──────────────────────────────────────────────────────
    // [의사결정 2] 로그인 경로
    // ──────────────────────────────────────────────────────
    
    Integer currentOrder = Integer.MAX_VALUE;  // ← 기본값: 모두 활성화
    
    // 심사 스테이지가 있는 그룹?
    Integer pointStageCheck = mapQueryMapper.pointStageCheck(trackNo, sgNo);
    
    if (pointStageCheck > 0) {
        
        log.info("[Stage List] 심사 있음 경로: trackNo={}, sgNo={}",
                 trackNo, sgNo);
        
        // ──────────────────────────────────────────────────────
        // 심사가 있으면: 신청 필수, 합격 여부에 따라 결정
        // ──────────────────────────────────────────────────────
        
        // 신청 스테이지가 있는가?
        Integer applyStageCount = mapQueryMapper
            .applyStageCount(trackNo, sgNo);
        
        if (applyStageCount == 0) {
            // 신청 스테이지 없음 → 이 그룹은 대상자 확인 필요
            Integer pointStageTargetCount = mapQueryMapper
                .pointStageTargetCount(trackNo, sgNo, userNo);
            
            if (pointStageTargetCount == null || pointStageTargetCount < 1) {
                // 이 그룹의 심사 대상자가 아님 → 버튼 모두 비활성
                currentOrder = 0;
            }
        } else {
            // 신청 스테이지 있음 → 합격 여부에 따라 결정
            currentOrder = currentOrderToPointStage(
                trackNo, sgNo, userNo, currentOrder);
        }
        
    } else {
        
        log.info("[Stage List] 심사 없음 경로: trackNo={}, sgNo={}",
                 trackNo, sgNo);
        
        // ──────────────────────────────────────────────────────
        // 심사가 없으면: 이전 그룹의 심사 합격 여부 확인
        // ──────────────────────────────────────────────────────
        
        Integer groupRoleCheck = mapQueryMapper
            .groupRoleCheck(trackNo, sgNo, userNo);
        
        if (groupRoleCheck == null || groupRoleCheck < 1) {
            // 이전 그룹에서 합격하지 못함 → 이 그룹 비활성
            currentOrder = 0;
        }
        // else: MAX 유지 (모두 활성화)
    }

    // ──────────────────────────────────────────────────────
    // [최종 처리] currentOrder 기준으로 버튼 활성화
    // ──────────────────────────────────────────────────────
    
    log.info("[Stage List] 최종 currentOrder={}", currentOrder);
    
    for (StageListInfoVO stage : stageList) {
        // Stage별 버튼 조회 (currentOrder까지만)
        ArrayList<StageButtonListVO> buttonList = mapQueryMapper
            .getApplyTrackButtonList(stage.getStageNo(), currentOrder, userNo);
        
        // ──────────────────────────────────────────────────────
        // 버튼별 제출 상태 조회 (sbt_no별로 다름)
        // ──────────────────────────────────────────────────────
        
        for (StageButtonListVO button : buttonList) {
            switch (button.getSbtNo().intValue()) {
                case 3:  // 공지/심사공지
                    button.setSubmitState(
                        mapQueryMapper.getNoticeSubmissionState(
                            userNo, trackNo, stage.getStageNo()));
                    break;
                    
                case 5:  // 신청/심사결과
                    resolveApplyButtonState(button, userNo, trackNo);
                    break;
                    
                case 9:  // 자료제출
                    button.setSubmitState(
                        mapQueryMapper.getSubmissionState(
                            userNo, trackNo, stage.getStageNo()));
                    break;
                    
                case 7:  // 미비서류
                    button.setSubmitState(
                        mapQueryMapper.getSubmissionIncompleteState(
                            userNo, trackNo, stage.getStageNo()));
                    break;
            }
        }
        
        stage.setStageButtonList(buttonList);
    }
    
    return stageList;
}

// ──────────────────────────────────────────────────────
// [Helper] 심사 스테이지에서 합격 여부 → currentOrder 결정
// ──────────────────────────────────────────────────────

private Integer currentOrderToPointStage(
        Long trackNo, Long sgNo, Long userNo, Integer defaultOrder) {
    
    // 이 그룹의 모든 심사 스테이지를 합격 여부와 함께 조회
    List<StageShowInfoVO> pointStageList = mapQueryMapper
        .pointStagePassList(trackNo, sgNo, userNo);
    
    // 합격하지 못한 첫 번째 스테이지의 order를 한계로 설정
    for (StageShowInfoVO stage : pointStageList) {
        if (stage.getPtPass() == null || stage.getPtPass() == 0) {
            // 불합격 찾음 → 이전 order까지만 활성화
            return stage.getStageOrder();
        }
    }
    
    // 모든 심사에서 합격 → 다음 단계 전부 활성화
    return defaultOrder;
}
```

## ⚠️ 마주친 문제점 & 해결

### **Problem 1: 분기의 폭발**

**증상:**
```java
// 나쁜 예: if-else 폭발
if (notLogin) {
    if (hasApply) {
        if (hasPass) {
            if (hasPoint) {
                // ...
            }
        }
    }
} else if (login) {
    // ...
}

→ 깊이 10단계 이상, 버그 가능성 ❌
```

**해결:**
```java
// 좋은 예: currentOrder로 추상화
Integer currentOrder = calculateCurrentOrder(...);

for (stage : stageList) {
    stage.setButtons(getButtonsByOrder(stage, currentOrder));
}

→ 명확한 흐름, 유지보수 쉬움 ✅
```

### **Problem 2: 버튼별 제출 상태 쿼리 중복**

**증상:**
```
각 버튼(sbt_no=3,5,7,9)마다 다른 쿼리
→ 4개 버튼 × N개 스테이지 = N+1 폭발 ❌
```

**해결:**
```java
// switch로 sbt_no별 분기
switch (button.getSbtNo()) {
    case 3 -> getNoticeState(...);
    case 5 -> getApplyState(...);
    // ...
}

// 모든 버튼을 1회 루프에서 처리 ✅
```

### **Problem 3: 심사 스테이지 여부 놓침**

**증상:**
```
심사가 없는 그룹에서도 심사 합격 로직 실행
→ 불필요한 쿼리, 틀린 결과 ❌
```

**해결:**
```java
Integer pointStageCheck = mapQueryMapper.pointStageCheck(...);

if (pointStageCheck > 0) {
    // 심사 로직
} else {
    // 비심사 로직
}

→ 명확한 분기 ✅
```

## 📊 성과

### Before

```
분기 규칙이 코드에 흩어짐:
├─ Controller에도 if-else
├─ Service에도 if-else
├─ Mapper에서도 분기
└─ 유지보수 어려움 ❌

버튼 활성화 오류:
├─ 활성화되면 안 될 버튼이 활성화
├─ 사용자가 잘못된 작업 시도
└─ 데이터 불일치 ❌
```

### After

```
currentOrder 하나로 모든 활성화 결정:
├─ 명확한 단일 책임
├─ 유지보수 쉬움 ✅
└─ 버그 가능성 낮음 ✅

버튼 활성화 정확:
├─ 상태에 맞는 버튼만 활성화
├─ 사용자 실수 방지 ✅
└─ 데이터 일관성 보장 ✅
```

## 🎓 배운 점

### 1. 복잡한 분기는 한 변수로 추상화

> **"모든 결정을 한 값(currentOrder)으로 표현하라"**

```
❌ if-else 깊이
✅ 단일 정수값 (0, order, MAX)
```

### 2. 의사결정 트리를 주석으로 명시

```java
// [비로그인?]
//   [신청했나?]
//     [심사 있나?]
//       [합격?]
```

→ 같은 개발자도 1개월 뒤 이해 가능

### 3. 버튼별 로직은 switch로

```java
switch (sbtNo) {
    case 3 -> ...
    case 5 -> ...
}
```

→ 새 버튼 추가 시 case만 추가
