## 🎯 문제 정의

### 상황: 팀원 인증

```
1. 팀원 등록 (admin)
   └─ state = 3 (가등록)
   └─ tm_token = "abc123xyz..." 생성
   └─ 이메일 발송: "이 링크를 클릭하세요 →
      /api/user/apply/command/confirm?token=abc123xyz..."

2. 팀원이 이메일 링크 클릭
   ├─ 브라우저: GET /api/user/apply/command/confirm?token=abc123xyz
   └─ 백엔드: state = 3 → 0으로 변경 ✅

3. 하지만...
   ├─ 팀원이 같은 링크를 또 클릭? (멱등성)
   ├─ 여러 팀원이 동시 인증? (동시성)
   └─ 잘못된 토큰 입력? (검증)
```

### 복잡성

```
❌ 문제 1: 여러 번 클릭해도 안전한가? (멱등성)

첫 클릭:
UPDATE team_member SET state = 0 WHERE tm_token = 'abc123'
→ state = 3 → 0 (변경됨) ✅

두 번째 클릭:
UPDATE team_member SET state = 0 WHERE tm_token = 'abc123'
→ state = 0 → 0 (변경 없음)
→ 이미 인증된 팀원인데 다시 처리? ⚠️

❌ 문제 2: 여러 팀원 동시 인증 안전?

팀원 A: state = 3, token = 'abc123'
팀원 B: state = 3, token = 'def456'

동시에 인증 클릭:
├─ Tx1: UPDATE WHERE token = 'abc123' → state = 0
├─ Tx2: UPDATE WHERE token = 'def456' → state = 0
└─ 동시성 이슈? (MySQL은 원자적 UPDATE)

❌ 문제 3: 토큰 검증

토큰이 존재하지 않으면?
└─ UPDATE의 영향 행 = 0
└─ 이미 인증했는지? 잘못된 토큰인지? 구분 불가 ⚠️
```

## 💡 솔루션: WHERE 조건에 state=3 포함

```
핵심: UPDATE의 WHERE 조건에 "state=3" 추가

UPDATE team_member
SET state = 0
WHERE tm_token = ?
  AND state = 3      ← 이 조건!

효과:
├─ 첫 클릭: state=3 → state=0 (매칭됨, 변경) ✅
├─ 둘째 클릭: state=0 (매칭 안 됨, 변경 없음) → 멱등성 ✅
├─ 동시 인증: 각각 다른 token → 각각 다른 행 → 안전 ✅
└─ 잘못된 토큰: 매칭되는 행 없음 → affectedRows=0
```

## 🔥 핵심 구현 코드

### Controller: 인증 엔드포인트

```java
@Operation(summary = "팀원 인증 API", 
           description = "이메일 토큰으로 팀원을 가등록 → 정상 상태로 전환")
@NoneAuth  // ← 인증 없이 접근 가능 (이메일 링크에서 진입)
@GetMapping("/confirm")
public ResponseData confirmTeamMember(
        @RequestParam("token") String token) {
    
    log.info("[Confirm Member] token 인증 시작: {}", 
             token.substring(0, Math.min(10, token.length())) + "...");
    
    try {
        applyCommandService.updateStateToActive(token);
        
        return ResponseData.builder()
            .message("팀원 인증이 정상적으로 완료되었습니다")
            .code(HttpStatus.OK.value())
            .build();
        
    } catch (OperationErrorException e) {
        
        // 이미 인증했거나 토큰 오류
        log.warn("[Confirm Member] 토큰 인증 실패: {}", e.getMessage());
        
        return ResponseData.builder()
            .message("유효하지 않은 또는 이미 처리된 토큰입니다")
            .code(HttpStatus.BAD_REQUEST.value())
            .build();
    }
}
```

### Service: 멱등 UPDATE 로직

```java
@Transactional
public void updateStateToActive(String token) {
    
    // ──────────────────────────────────────────────────────
    // [검증] 토큰이 NULL이거나 빈 문자열인가?
    // ──────────────────────────────────────────────────────
    
    if (token == null || token.trim().isEmpty()) {
        throw new OperationErrorException(
            ErrorCode.INVALID_TOKEN,
            "토큰이 제공되지 않았습니다");
    }
    
    log.info("[Update State] token 기반 UPDATE 시작");
    
    // ──────────────────────────────────────────────────────
    // [핵심] WHERE 조건에 "state = 3" 포함 (멱등성 보장)
    // ──────────────────────────────────────────────────────
    
    int affectedRows = applyCommandMapper.updateMemberStatus(token);
    
    // ──────────────────────────────────────────────────────
    // [검증] 영향받은 행이 0이면 실패
    // ──────────────────────────────────────────────────────
    
    if (affectedRows < 1) {
        log.warn("[Update State] 토큰 매칭 실패. affectedRows={}", affectedRows);
        
        throw new OperationErrorException(
            ErrorCode.INVALID_TOKEN,
            "해당 토큰을 찾을 수 없거나 이미 처리되었습니다");
    }
    
    log.info("[Update State] 팀원 상태 업데이트 완료: {} 행 변경됨", 
             affectedRows);
}
```

### MyBatis: 멱등 UPDATE 쿼리

```xml
<!-- ApplyCommandMapper.xml:1094 -->
```

## 📊 실행 흐름

### Scenario 1: 정상 인증 (첫 클릭)

```
[Before]
team_member:
├─ tm_no: 456
├─ state: 3 (가등록)
├─ tm_token: "abc123xyz"
└─ tm_name: "김철수"

[Request]
GET /api/user/apply/command/confirm?token=abc123xyz

[Query 실행]
UPDATE team_member
SET state = 0, tm_token = NULL
WHERE tm_token = 'abc123xyz' AND state = 3
      └─ 매칭됨! ✅

[After]
├─ state: 0 (정상)
├─ tm_token: NULL
└─ Response: "인증 완료" ✅
```

### Scenario 2: 중복 클릭 (멱등성)

```
[첫 클릭 후 상태]
team_member:
├─ state: 0 (이미 정상)
├─ tm_token: NULL

[두 번째 클릭]
GET /api/user/apply/command/confirm?token=abc123xyz

[Query 실행]
UPDATE team_member
SET state = 0, tm_token = NULL
WHERE tm_token = 'abc123xyz' AND state = 3
      └─ 매칭 안 됨 (state=0이라서) ❌

[결과]
affectedRows = 0
└─ Exception: "유효하지 않은 토큰"
└─ 하지만 UI는 "이미 처리됨" 메시지로 친화적 ✅
```

### Scenario 3: 동시 인증 (동시성)

```
팀원 A: token='abc', state=3
팀원 B: token='def', state=3

[동시 요청]
├─ Tx1: GET /confirm?token=abc
├─ Tx2: GET /confirm?token=def

[각각 실행]
Tx1: UPDATE ... WHERE token='abc' AND state=3 → 팀원 A만 변경 ✅
Tx2: UPDATE ... WHERE token='def' AND state=3 → 팀원 B만 변경 ✅

[결과]
둘 다 성공 (서로 영향 없음) ✅
```

## ⚠️ 마주친 문제점 & 해결

### **Problem 1: 멱등성 미보장**

**증상:**
```sql
-- 나쁜 예
UPDATE team_member SET state = 0 WHERE tm_token = ?

첫 클릭: state = 3 → 0 ✅
둘째 클릭: state = 0 → 0? (이미 0인데 다시 처리) ⚠️

로그에 "인증 완료"가 2번 기록됨 → 감시 로그 오염
```

**해결:**
```sql
-- 좋은 예
UPDATE team_member 
SET state = 0 
WHERE tm_token = ? 
  AND state = 3  ← 이 조건으로 "첫 번째만" 매칭

결과:
├─ 첫 클릭: 1행 변경
├─ 둘째 클릭: 0행 변경 → Exception
└─ 멱등성 보장 ✅
```

### **Problem 2: 토큰 유효성 불분명**

**증상:**
```
affectedRows = 0이 나올 수 있는 경우:
├─ 이미 인증한 팀원
├─ 잘못된 토큰
├─ 토큰 만료
└─ 모두 다른 상황인데 같은 에러 메시지

사용자: "뭐가 문제야?" 😕
```

**해결:**
```java
// Controller에서 친화적 메시지
if (affectedRows < 1) {
    return "유효하지 않은 또는 이미 처리된 토큰입니다"
    // → 사용자가 이해 가능
}
```

### **Problem 3: 토큰 재사용**

**증상:**
```
토큰이 남아있으면:
├─ 팀원 A가 인증했지만 토큰 still 있음
├─ 다른 사람이 같은 토큰으로 다시 인증? 😱
```

**해결:**
```sql
UPDATE team_member
SET tm_token = NULL,  ← 토큰 즉시 제거
    state = 0
WHERE tm_token = ? AND state = 3

→ 토큰은 1회용이 됨 ✅
```

## 📊 성과

### Before (상태 조건 없음)

```
UPDATE team_member SET state = 0 WHERE tm_token = ?

문제:
├─ 중복 클릭 → 로그 오염
├─ affectedRows로 성공/실패 구분 불명확
└─ 토큰 재사용 위험
```

### After (WHERE state=3 추가)

```
UPDATE team_member SET state = 0 WHERE tm_token = ? AND state = 3

성과:
├─ 멱등성: 여러 번 클릭 안전 ✅
├─ 명확한 상태: affectedRows로 1회 인증만 확인 ✅
├─ 토큰 제거: 즉시 무효화 ✅
└─ 동시성: 여러 팀원 동시 인증 안전 ✅
```

## 🎓 배운 점

### 1. UPDATE의 WHERE 조건으로 멱등성 구현

> **"상태 조건을 WHERE에 넣으면 멱등성이 자동으로 따라온다"**

```
❌ UPDATE SET state = ? WHERE id = ?
   → 몇 번 실행해도 같은 값

✅ UPDATE SET state = 0 WHERE id = ? AND state = 3
   → 첫 번째만 실행, 두 번째는 0행 변경
```

### 2. 외부 진입점은 @NoneAuth 활용

> **"이메일 링크는 JWT 토큰이 없으니 인증 생략"**

```java
@NoneAuth
@GetMapping("/confirm")
public ResponseData confirmTeamMember(@RequestParam String token) {
    // 토큰만으로 검증
}
```

### 3. affectedRows를 유효성 검사로

```java
if (affectedRows < 1) {
    throw Exception;  // 토큰 오류
}
// → 성공: affectedRows >= 1
```
